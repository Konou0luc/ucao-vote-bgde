# PROJECT OVERVIEW
━━━━━━━━━━━━━━━━━━━━━━

The platform allows:
- student authentication using matricule + OTP email;
- secure anonymous voting;
- prevention of double voting;
- admin dashboard;
- election management;
- participation tracking;
- automatic vote counting;
- audit logging;
- result generation.

Main constraint:
The system MUST guarantee anonymity.

There must NEVER be a direct link between:
- the student identity;
- and the selected vote.

Participation and votes must remain separated.

# SECURITY REQUIREMENTS
━━━━━━━━━━━━━━━━━━━━━━

Security is CRITICAL for this project.

You must implement:
- JWT authentication;
- bcrypt password hashing;
- OTP hashing;
- rate limiting;
- helmet;
- cors whitelist;
- environment variables;
- centralized error handling;
- audit logging;
- strict validation;
- secure session flow;
- anti double-vote protection.

The vote MUST remain anonymous at all times.

IMPORTANT:
There must NEVER be a direct relation between:
- Student
- Vote

Participation and votes must stay technically separated.

OTP rules:
- 6 digits;
- hashed in database;
- expires after 10 minutes;
- maximum 3 attempts;
- invalidated after usage;
- resend OTP allowed.

# BUSINESS RULES
━━━━━━━━━━━━━━━━━━━━━━

Students:
- authenticate with matricule;
- confirm identity;
- receive OTP by email;
- validate OTP;
- vote once only.

Voting:
- anonymous voting;
- definitive vote;
- no vote modification;
- no double vote.

Admins:
- secure JWT login;
- manage elections;
- manage candidate lists;
- monitor participation;
- access results;
- access audit logs.

Audit logs:
- record sensitive actions;
- NEVER store vote choices.
